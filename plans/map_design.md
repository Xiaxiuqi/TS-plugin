# 沉浸式层级地图系统设计方案 (v1.5.0)

## 1. 核心概念

目标是创建一个类似 RPG 游戏（如崩坏：星穹铁道）的地图系统，支持宏观区域概览和微观室内详情的无缝切换。

### 1.1 层级架构

* **宏观层 (Macro Layer)**: 现有的地图模式。显示区域节点（如“酒吧”、“街道”、“公园”）。
  * **表现**: 矩形/圆形节点，拓扑连接。
  * **交互**: 点击进入微观层（如果存在）。
* **微观层 (Micro Layer)**: 具体的室内/局部场景布局。
  * **表现**: 顶视图。包含墙壁（不可通行）、家具（障碍物/交互点）、地板（可通行）、装饰。
  * **动态性**: AI 根据剧情文本实时生成或更新布局。

## 2. 数据结构设计

为了支持微观地图，我们需要扩展现有的表格结构。

### 2.1 主要地点表 (Main Location Table) - 扩展

保持现有结构，但增加一个字段用于标记或存储微观数据的索引。

| 列名           | 说明                                                                     |
| :------------- | :----------------------------------------------------------------------- |
| 地点名称       | 唯一标识 (Key)                                                           |
| ...            | (现有字段: 坐标, 宽高, 描述)                                             |
| **微观布局ID** | (新增) 关联到`场景详情表`的ID，或者直接存储简单的布局JSON (如果长度允许) |

### 2.2 场景详情表 (Scene Detail Table) - 新增

这是一个专门用于存储微观布局数据的表格。AI 将根据 prompt 将自然语言描述转化为结构化数据填入此表。

| 列名       | 说明                                                | 示例值                                                                   |
| :--------- | :-------------------------------------------------- | :----------------------------------------------------------------------- |
| 场景ID     | 对应主要地点表的地点名称                            | 酒吧                                                                     |
| 墙壁数据   | 定义不可通行的墙体轮廓 (SVG Path d属性或坐标点序列) | `M0,0 L100,0 L100,100 L0,100 Z` (围墙)                                   |
| 家具布局   | 定义内部物体 (类型, 坐标, 尺寸, 旋转, 交互ID)       | `[{"type":"bar_counter","x":10,"y":10,"w":80,"h":20,"interact":"酒柜"}]` |
| 区域元数据 | 场景的大小、入口位置等                              | `{"width":800,"height":600,"entry":[50,590]}`                            |
| 状态哈希   | 用于检测描述是否变化，决定是否重绘                  | (Hash of description)                                                    |

### 2.3 地图元素表 (Map Element Table) - 关联

现有的`地图元素表`继续用于定义可交互物体（NPC, 物品）。

* **关联**: 元素的 `所属主地点` 对应微观场景。
* **坐标**: 元素的 `X坐标`, `Y坐标` 将直接映射到微观地图的坐标系中。

## 3. 提示词策略 (System Prompt)

需要设计一个专门的 Function Calling 或结构化输出指令，让 AI 完成 "Text to Layout" 的转换。

### 3.1 坐标系定义

* **画布**: 设定标准微观画布为 800x600 (单位：像素)。
* **原点**: 左上角 (0,0)。
* **方向**: X轴向右，Y轴向下。

### 3.2 转换逻辑 (Prompt 示例)

```markdown
你是一个室内设计师AI。请根据以下场景描述，生成微观地图布局数据。
场景："{context_description}"

要求：
1. 识别墙壁结构，生成封闭的多边形坐标列表。
2. 识别家具/设施，估算其在顶视图中的位置(x,y)和尺寸(w,h)。
3. 家具类型映射到标准图元：[table, chair, bed, counter, cabinet, door, decor]。
4. 输出格式为 JSON。
```

## 4. 前端渲染逻辑 (Frontend)

### 4.1 渲染引擎升级 (`renderMap`)

* **状态管理**: 新增 `currentViewMode` ('macro' | 'micro') 和 `currentSceneId`。
* **SVG 绘制**:
  * **墙壁**: 使用 `<path>` 绘制，填充深色/纹理，描边高亮。
  * **家具**:
    * 通用矩形/圆形: `<rect>`, `<circle>`。
    * 预制图标/SVG: 使用 `<use>` 引用预定义的家具符号（如椅子、床的简笔画）。
  * **角色**: 在微观坐标中绘制角色头像。

### 4.2 交互逻辑

* **进入**: 点击宏观地图的节点 -> 检查是否有微观数据 -> 切换视图。
* **返回**: 点击“返回上层”按钮 -> 切换回宏观视图。
* **点击元素**: 点击家具/物品 -> 弹出交互菜单 (复用现有的 `openMapElementModal`)。

## 5. 开发计划

1. **模板更新**: 创建包含 `场景详情表` 的新 JSON 模板。
2. **Prompt 编写**: 编写用于生成布局数据的 System Prompt。
3. **前端重构**:
    * 修改 `index.ts` 中的 `renderMap` 函数。
    * 增加微观地图的渲染子函数 `renderMicroMap`。
    * 增加 SVG 图元库 (简单的家具顶视图)。
4. **联调**: 测试 AI 生成数据的准确性和前端渲染的兼容性。
