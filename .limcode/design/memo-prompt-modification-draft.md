> 当前基准备份：`src/数据库前端/奶龙王+奶龙工具箱/奶龙工具箱/docs/archive/备忘录表提示词基准-2026-05-22.md`

## 1. 本轮修改目标

本轮主要解决以下问题：

1. 长剧情中大量阶段节点一直保持“未归档”，导致旧推理、旧假设、旧阶段和当前有效信息混在一起。
2. 阶段性假设被写成最终结论，后续剧情推翻后仍留在未归档记录中。
3. 备忘录内容仍有纪要化倾向，容易把证据细节压缩成“逻辑闭环”“动机升级”“定性为……”等概括句。
4. 示例必须完全自造，不能使用用户提供的测试案例、人物、地点、组织、能力、案件结构、证据链或原话结构。

## 2. 拟新增规则一：假设 / 证据 / 定论分层

建议插入位置：

- 放在【细节保真原则】之后；或
- 放在【核心原则：阶段颗粒度与防增殖】之前。

拟新增文本：

【假设、证据、定论分层】

长期事件中的推理必须区分“假设”“证据”“定论”，不得把阶段性推测写成最终结论。

1. 假设：只有单条线索、单次推理、单一角色判断、尚未交叉验证时，只能写“当前推测”“阶段性假设”“待验证方向”。禁止写“确认”“证实”“定性为”“彻底闭环”。
2. 证据：必须写清楚证据来源和证据细节，例如日期、时间、金额、文档名、记录编号、申请理由、联系人、职位、定位时间、流水方向、物理痕迹、原话原文等。
3. 定论：只有两类以上独立证据互相支持，且暂无反证时，才可写“初步确认”；只有完整证据链形成且后续没有明显反证时，才可写“确认”“证实”“定性”。
4. 修正：若后续证据推翻旧假设，必须处理旧备忘录。旧条目不得继续保持“未归档”并当作有效结论。
5. 写法：记录内容应按“证据细节 → 反常点 → 当前推断/定论等级”的顺序书写。

禁止示例：

- `案件动机定性为绝望掠夺。`
- `证据链彻底闭环，确认其为主谋。`
- `逻辑已补全，调查转向幕后黑手。`

推荐示例：

- `当前仅有借阅卡刷卡时间与书库门禁重合，暂记为“内外协助假设”，需等待监控盲区复核。`
- `因《古籍修复申请单》、502号借阅卡、21:18门禁记录三者互相吻合，可初步确认有人借修复流程进入闭架书库。`

## 3. 拟新增规则二：阶段状态收束规则

建议插入位置：

- 放在【归档状态规则】之后；或
- 放在【更新触发 (Update)】开头。

拟新增文本：

【阶段状态收束规则】

同一长期事件中，AI 每次写入前必须检查已有未归档阶段节点，避免旧阶段长期堆积。

1. 当前阶段仍在持续补充：保持“未归档”，并更新该阶段记录。
2. 当前阶段已经结束，且后续阶段已经开始：将旧阶段更新为“已归档”。
3. 旧阶段的判断被后续证据推翻：将旧阶段更新为“已废除”；若数据库允许备注字段，可注明“被后续证据修正”。
4. 旧阶段被后续更完整记录吸收：将旧阶段更新为“已归档”或“已废除”，禁止继续保持“未归档”。
5. 同一长期事件中，未归档阶段节点应只保留当前仍需继续补充的记录；禁止把所有历史阶段都长期保留为“未归档”。
6. 若旧条目包含仍有价值的原始证据，但推论被推翻，应保留证据细节，废除或修正推论，不得删除证据。

处理优先级：

1. 先处理被推翻的旧假设。
2. 再归档已经完成的旧阶段。
3. 最后新增或更新当前阶段。

## 4. 拟新增规则三：证据优先写法

建议插入位置：

- 替换或强化现有【细节保真原则】；
- 同时在【更新触发 (Update)】中追加。

拟新增文本：

【证据优先写法】

备忘录记录内容必须优先保存证据，而不是优先总结结论。

记录顺序推荐：

1. 证据来源：谁提供、从哪里查到、哪个文档/工具/记录。
2. 证据细节：日期、时间、金额、数量、重量、距离、姓名、职位、公司名、文档名、原话原文、物理痕迹。
3. 反常点：正常情况应如何，实际情况如何，两者差异是什么。
4. 当前判断：只能根据证据强度写“待验证 / 阶段性假设 / 初步确认 / 确认”。

禁止只写：

- `闭环了经济动机。`
- `锁定共犯。`
- `案件性质升级。`
- `推翻旧判断。`

必须写成：

- `某日某时、某文档、某金额、某对象、某物理痕迹共同支持什么判断。`

## 5. 拟新增规则四：更新触发中的旧条目处理

建议插入位置：

- `更新触发 (Update):` 内。

拟新增文本：

更新触发时，AI 不得只看当前要写入的内容，还必须检查同事件链已有未归档条目。

若新证据与旧条目存在冲突：

1. 旧条目是阶段性假设：更新其归档状态为“已废除”，或将内容改为“该推断已被后续证据修正”。
2. 旧条目是证据记录：保留证据，删除或修正过时推论。
3. 旧条目是已完成阶段：更新为“已归档”。
4. 新条目不得在旧条目未处理时直接新增，避免同一事件链出现多个互相冲突的未归档判断。

SQL 示例：

-- 示例：废除被后续证据推翻的阶段性假设
UPDATE memo
SET archive_status = '已废除'
WHERE title = '馆藏失窃：外来访客假设'
  AND category = '阶段节点类'
  AND archive_status = '未归档';

-- 示例：归档已经完成的旧阶段
UPDATE memo
SET archive_status = '已归档'
WHERE title = '馆藏失窃：闭架书库门禁核查'
  AND category = '阶段节点类'
  AND archive_status = '未归档';

## 6. 拟新增规则五：完全无关示例体系

后续示例建议使用“城市图书馆失窃案”，避免与用户测试案例相似。

该示例体系不使用：

- 用户测试案例的人物名；
- 用户测试案例的组织名；
- 用户测试案例的地点类型；
- 用户测试案例的能力体系；
- 用户测试案例的案件结构；
- 用户测试案例的经济/医疗/物流/仓库/违禁品证据链。

自造示例实体：

- 事件链：馆藏失窃
- 地点：市立图书馆闭架书库
- 人物：主角、馆员许澄、修复师孟临、保安赵砚
- 物件：502号借阅卡、蓝封修复胶、闭架书库门禁、温湿度记录仪、缺页古籍
- 关键证据：21:18门禁记录、空白修复申请单、修复胶残留、恒湿柜异常开合记录

示例目标：

1. 展示阶段节点如何拆分。
2. 展示秘密线索如何保留原文/证据细节。
3. 展示假设如何被后续证据修正。
4. 展示旧条目如何归档或废除。
5. 保留盲盒多子项成果示例，因为这是通用需求。
