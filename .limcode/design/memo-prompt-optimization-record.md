# 备忘录表提示词优化项目文档

## 1. 文档目的

本文件用于记录“备忘录表提示词”从原始版本到后续优化方向的需求、问题、修改原因与设计约束。

当前阶段只做项目记录与修改计划，不直接修改业务提示词文件。

目标文件：

```text
src/数据库前端/奶龙王+奶龙工具箱/奶龙工具箱/docs/备忘录表合并更新提示词.md
```

## 2. 原始提示词基线

原始提示词的核心结构为数据库模板标准结构：

```text
表格说明 (Note):
初始化触发 (Init):
新增触发 (Insert):
更新触发 (Update):
删除触发 (Delete):
DDL 定义 (SQLite 模式)
CREATE TABLE 语句:
```

原始表字段为：

1. 标题
2. 分类
3. 归档状态
4. 记录日期
5. 记录内容
6. 涉及对象
7. 关键词

原始分类为三类：

```text
秘密线索类
大事件类
约定承诺类
```

原始核心原则：

- 禁止流水账。
- 连续性、多阶段事件不能每个小步骤都新增。
- 事件闭环后汇总为一条高密度备忘录。
- 多子项成果事件，如“研发一套10只盲盒”，必须逐项列出名称、设计、功能、数量等关键属性。
- 秘密线索类与约定承诺类不可更新正文。
- 大事件类可在未归档时更新。
- 已归档 / 已废除后禁止更新。
- 关键词必须是具体名词，不能使用泛化地点名或泛化名词。

原始 DDL 约束大致为：

```sql
CREATE TABLE memo (
  row_id INTEGER PRIMARY KEY,
  title TEXT NOT NULL,
  category TEXT NOT NULL CHECK(category IN ('秘密线索类', '大事件类', '约定承诺类')),
  archive_status TEXT NOT NULL DEFAULT '未归档' CHECK(archive_status IN ('未归档', '已归档', '已废除')),
  record_date TEXT NOT NULL,
  content TEXT NOT NULL CHECK(LENGTH(content) <= 400),
  involved_objects TEXT,
  keywords TEXT
);
```

## 3. 已观察到的问题

### 3.1 过度新增导致条目增殖

早期问题是 AI 对连续调查链的每一个小步骤都新增一条备忘录，例如：

- 初识组织与邀请
- 接头与能力初测
- 获得残响
- 调查邀请
- 推演动机
- 抵达现场
- 现场感官矛盾

这类记录导致备忘录变成流水账，召回时噪声过多。

### 3.2 过度合并导致细节丢失

后续加入“合并”规则后，又出现相反问题：AI 把长期案件中的多个阶段压缩成少量大段概括，导致备忘录失去“纪要表的细节补充”作用。

典型失败：

- 原始关键线索应保留原话、感官细节、反常表现。
- AI 却写成“获得关键线索”“逻辑闭环”“核心违和点”等纪要式表达。
- 例如原本应记录“感官回溯中捕捉到临死前微弱求保密讯息，且受害者在暴力和溺水前表现出非典型顺从与保护欲”，却被压缩为“获得遗言并推演保护逻辑”。

### 3.3 阶段节点仍会无限延展

长剧情测试中出现了大量连续未归档阶段节点：

- 案件动机、共犯、经济流向、赌债、病情、黑仓、违禁品、上游公司等内容连续新增。
- 其中部分节点后续被推翻，但仍保持未归档。
- 例如“生命末期的绝望掠夺”被后续黑仓 / 违禁品链条修正，但旧条目没有被归档、废除或更新为“阶段性误判”。

问题本质：提示词缺少“假设 / 证据 / 定论”的分层管理。

### 3.4 关键词不利于注入和召回

出现了把标题前缀作为关键词的情况，例如长期事件名被反复放入关键词。

用户反馈：标题式大词作为关键词难以用于精准注入，关键词应更偏向具体证据、物品、人物、地点、能力、文档、金额、日期等可检索实体。

注意：用户后续表示“标题变成备忘录关键词这个东西不必再修改了”，因此本问题只记录，不作为当前优先修改目标，除非后续再次提出。

### 3.5 示例污染测试

用户多次强调：提供的实际填表内容都是测试案例，不允许直接放进提示词示例。

失败点：曾把用户案例中的组织、地点、能力、残响、仓库等结构改名后继续作为示例，仍会导致测试失真。

后续所有示例必须使用完全无关的自造剧情，不得沿用用户案例结构。

## 4. 当前明确需求

### 4.1 备忘录定位

备忘录表是纪要表的细节补充。

- 纪要表：负责概括。
- 备忘录表：负责保存具体证据、原话、原文、感官细节、物理痕迹、关键对象、阶段性误判与修正。

### 4.2 颗粒度目标

备忘录不能：

1. 每次回复 / 每个小动作新增一行。
2. 永远只更新同一行，导致细节被概括。
3. 把阶段性假设写成最终定论。
4. 放任被后续推翻的旧条目继续未归档。

备忘录应该：

1. 阶段内合并。
2. 阶段间允许拆分。
3. 关键线索允许独立成条。
4. 假设和定论必须区分。
5. 被推翻、修正、吸收的阶段性判断应及时归档或废除。

### 4.3 分类方向

当前用户接受五项分类：

```text
秘密线索类
阶段节点类
大事件类
多子项成果类
约定承诺类
```

其中需要重点强化：

- 阶段节点类：记录可独立召回的阶段细节。
- 秘密线索类：可用于保留关键原话、原文、感官线索，不强制并入阶段节点。
- 大事件类：更适合闭环转折或已确认结果。
- 多子项成果类：保留盲盒这类多子项事件的逐项细节。

### 4.4 关键词方向

当前关键词规则：

- 最多 5 个。
- 不要求写满。
- 禁止为了凑数加入泛化词。

用户指出标题式关键词不利于注入，但暂时说明“不必再修改”。后续若继续优化，可单独设计关键词规则。

## 5. 待解决关键设计点

### 5.1 假设、证据、定论分离

长案件中经常出现阶段性推理，例如：

- 经济动机假设。
- 情感动机假设。
- 绝望动机假设。
- 黑仓牟利假设。
- 违禁品转运假设。

这些不能直接写成最终定性。

需要新增规则：

- 未证实内容必须写成“阶段性假设 / 当前推断 / 待验证”。
- 后续证据推翻时，旧条目必须更新归档状态或内容中注明“已被后续证据修正”。
- 只有形成证据闭环时才可写成“确认 / 定性 / 证实”。

### 5.2 未归档数量控制

目前问题不是新增太多，而是大量阶段节点都保持“未归档”。

需要新增规则：

- 同一长期事件中，未归档阶段节点应只保留当前仍在推进、仍需继续补充的少数节点。
- 已完成阶段应设为“已归档”。
- 被推翻阶段应设为“已废除”。
- 被后续更完整记录吸收的阶段应归档或废除。

### 5.3 细节密度而非纪要式总结

需要继续强化：

- 记录内容先写证据和细节，再写推论。
- 禁止只有“闭环了某逻辑”“升级为某动机”“转向某调查”的抽象句。
- 若涉及数字、日期、金额、文档名、基站时间、公司名、职位名等，应尽量保留。

### 5.4 示例隔离

后续提示词示例必须完全自造，不得使用用户测试案例中的：

- 人物名；
- 组织名；
- 地点类型；
- 能力体系；
- 案件类型；
- 证据链结构；
- 道具功能；
- 原话结构。

## 6. 后续修改原则

1. 先更新项目文档和计划，再改提示词。
2. 改提示词时只使用自造示例。
3. 每次修改都记录：问题、原因、修改内容、预期效果。
4. 修改后需要人工确认再用于测试。
5. 不直接把用户提供的测试填表内容写进提示词示例。
