# 备忘录表提示词优化实施计划

## 0. 来源文档

本计划依据已创建的项目文档：

```text
.limcode/design/memo-prompt-optimization-record.md
```

该文档记录了原始提示词基线、当前失败案例、用户明确需求和后续修改原则。后续实施必须以该文档为准。

## 1. 当前目标

用户已经手动回档提示词，并明确要求：

1. 不直接修改提示词文件。
2. 先写项目文档记录原始提示词、每一步修改、原因和需求。
3. 再制定计划。
4. 后续若修改提示词，示例必须完全自造，不允许复用用户提供的测试案例。

因此，本阶段只规划，不改动目标提示词文件。

目标提示词文件为：

```text
src/数据库前端/奶龙王+奶龙工具箱/奶龙工具箱/docs/备忘录表合并更新提示词.md
```

## 2. 要解决的核心问题

### 2.1 阶段节点无限新增且长期未归档

当前测试中出现大量 `阶段节点类 / 未归档` 条目。它们记录了不同推理阶段，但后续剧情已经修正或推翻部分判断，旧条目仍然保持未归档。

需要新增规则：

- 同一长期事件中，未归档阶段节点不能无限堆积。
- 已经完成的阶段应归档。
- 被后续证据推翻的阶段应废除或更新为“阶段性误判”。
- 被后续更完整记录吸收的阶段应归档或废除。

### 2.2 阶段性假设被写成最终定论

测试中出现类似“案件动机定性为……”的表达，但后续剧情又出现新证据，导致该定性不再成立。

需要新增“假设 / 证据 / 定论”规则：

- 证据不足时，只能写“当前推测 / 阶段性假设 / 待验证方向”。
- 不得使用“定性为 / 彻底确认 / 证实 / 闭环”等最终表述，除非已经有完整证据链。
- 后续证据修正时，必须处理旧条目状态。

### 2.3 细节仍被纪要化

备忘录的定位是纪要表的细节补充，但测试中仍出现偏概括写法，例如只写“逻辑闭环”“动机升级”“调查目标转向”，而没有保留证据细节。

需要加强：

- 记录内容先写证据和细节，再写推理。
- 数字、日期、金额、文档名、职位、定位时间、公司名、具体原话等必须优先保留。
- 如果内容过长，不应概括掉证据，而应拆成可召回的阶段节点或秘密线索。

### 2.4 示例污染测试

用户提供的测试案例不能被写入提示词示例。即使改名，只要保留相似结构，也会导致测试失真。

后续所有示例必须满足：

- 不使用用户案例的人物名。
- 不使用用户案例的组织名。
- 不使用用户案例的地点类型。
- 不使用用户案例的能力体系。
- 不使用用户案例的案件类型。
- 不使用用户案例的证据链结构。
- 不使用用户案例的道具功能。
- 不使用用户案例的原话结构。

## 3. 后续提示词修改方向

### 3.1 增加“阶段状态收束规则”

拟新增规则：

```text
【阶段状态收束规则】
同一长期事件中，AI 每次写入前必须检查已有未归档阶段节点：
1. 若旧阶段已经完成且不再需要补充，更新为“已归档”。
2. 若旧阶段的判断被后续证据推翻，更新为“已废除”。
3. 若旧阶段被后续更完整阶段吸收，更新为“已归档”或“已废除”。
4. 若旧阶段仍在推进，才允许保持“未归档”。
5. 禁止同一事件链长期堆积大量未归档阶段节点。
```

### 3.2 增加“假设 / 证据 / 定论分层规则”

拟新增规则：

```text
【假设、证据、定论分层】
1. 只有单条线索或推理时，写“阶段性假设 / 当前推断 / 待验证”。
2. 有两类以上独立证据互相支持时，可写“高度疑似 / 初步确认”。
3. 有完整证据链且无明显反证时，才可写“确认 / 定性 / 证实”。
4. 后续证据推翻旧假设时，必须处理旧条目状态，不得继续保持未归档并当作有效结论。
```

### 3.3 增加“证据优先写法”

拟新增规则：

```text
【证据优先写法】
记录内容必须按“证据细节 → 反常点 → 当前推断”的顺序书写。
优先保留：
- 日期、时间、金额、数量、重量、距离；
- 文档名、报告名、申请理由、联系人、职位；
- 原话、原文、定位时间、流水方向；
- 物理痕迹、感官表现、工具或能力名称。
禁止只写结论而不写证据。
```

### 3.4 调整更新触发规则

拟强化：

- 更新时不能只改当前行，也要检查同事件链其他未归档行是否需要归档或废除。
- 若新证据修正旧动机、旧嫌疑方向、旧推论，应优先处理旧条目状态。
- 若旧条目只是阶段性推理，不应写成最终定论。

### 3.5 自造示例设计

后续示例将使用完全无关题材，避免与用户测试案例接近。

候选自造示例方向：

- 城市图书馆失窃案：围绕藏书章、借阅卡、修复胶水、馆内温湿度记录。
- 海边灯塔设备故障：围绕灯塔透镜、潮汐日志、旧电缆、巡检钥匙。
- 温室植物异常：围绕种子柜、温控阀、露水倒流、巡检册。

最终只选择一套，并确保不复用用户案例结构。

## 4. 计划执行步骤

### 步骤 1：确认项目文档

检查 `.limcode/design/memo-prompt-optimization-record.md` 是否准确记录：

- 原始提示词基线；
- 用户反馈的问题；
- 当前需求；
- 后续修改原则。

### 步骤 2：起草规则补丁

只起草，不直接写入目标提示词。

补丁内容包括：

1. 阶段状态收束规则。
2. 假设 / 证据 / 定论分层规则。
3. 证据优先写法。
4. 更新触发中的旧条目处理规则。
5. 示例隔离规则。

### 步骤 3：设计新示例体系

创建完全无关的自造示例，要求：

- 不贴近用户测试案例；
- 不使用类似的人物组织地点；
- 不使用类似能力体系；
- 不使用类似案件动机链；
- SQL 语法仍符合 SQLite。

### 步骤 4：提交草案给用户确认

在不修改目标提示词文件的前提下，向用户展示拟替换段落。

### 步骤 5：经用户确认后再修改提示词

用户确认后，才实际修改：

```text
src/数据库前端/奶龙王+奶龙工具箱/奶龙工具箱/docs/备忘录表合并更新提示词.md
```

### 步骤 6：修改后记录变更

更新项目文档，记录：

- 修改日期；
- 修改原因；
- 修改内容；
- 预期解决的问题；
- 待下一轮测试观察项。

## 5. 验收标准

修改后的提示词应满足：

1. 不再把阶段性假设写成最终定论。
2. 后续证据推翻旧判断时，会归档或废除旧条目。
3. 不再长期堆积大量未归档阶段节点。
4. 备忘录内容保留证据细节，而非纪要式概括。
5. 多子项成果仍逐项保留。
6. 关键词最多 5 个，不要求写满，不为凑数加入泛化词。
7. 示例不使用用户测试案例或相似结构。
8. SQL 示例保持 SQLite 语法有效。
