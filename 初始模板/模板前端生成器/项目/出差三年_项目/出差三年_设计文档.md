# 角色卡状态栏需求分析：出差三年（NTR调查版）

## 0. 技术可行性分析 (Technical Feasibility)

在开始设计之前，我已深入阅读并理解了 `模板前端生成器/范例/前端美化.html` 的核心逻辑：

1.  **数据通信机制**: 前端通过 `window.parent.AutoCardUpdaterAPI` 与宿主环境（如 SillyTavern）通信。
    *   **获取数据**: 使用 `dataSourceAPI.exportTableAsJson()` 主动拉取当前表格数据。
    *   **监听更新**: 使用 `dataSourceAPI.registerTableUpdateCallback(cb)` 注册回调，实现被动刷新。
    *   **发送指令**: 使用 `parent.triggerSlash('/setinput ' + msg)` 或 DOM 操作 `parent.document.getElementById('send_textarea')` 模拟用户输入，实现点击交互。

2.  **数据绑定原理**:
    *   后端（LLM）输出特定的 JSON 格式（`sheet_{UID}` 结构）。
    *   前端 JS 函数 `processJsonData(json)` 将复杂的 sheet 结构展平为易读的 `tables` 对象（Key 为表名）。
    *   渲染函数 `renderUI(tables)` 通过 `tables["表名"].rows[行索引][列索引]` 获取具体数值，并将其 `textContent` 或 `style.width` 应用到 HTML DOM 元素上。

基于此机制，我们可以构建一套完全动态、双向交互的“家庭监控系统”前端。

## 1. 核心概念与风格

*   **项目名称**: **家庭异变监控终端 (Home Anomaly Monitor)**
*   **核心玩法摘要**: 玩家扮演归家的丈夫，通过观察家中的细微变化（物品移动、衣物痕迹、妻女表情），在“维持家庭假象”与“揭露淫乱真相”之间博弈。核心机制是**“表象与真相的撕裂”**。
*   **UI视觉隐喻**: **“被黑客入侵的智能家居中控屏”**。
    *   表面上是高端、温馨的智能家居控制面板（显示温度、湿度、家庭成员位置）。
    *   实际上，界面上不断出现“系统错误”、“数据溢出”的故障特效，暗示家庭内部的崩坏。背景会有隐约的粉紫色噪点（暗示淫乱）和红色警告（暗示危险）。
*   **主色调**:
    *   **背景**: 深空灰/午夜蓝 (`#1a1b26`) —— 模拟暗黑模式的电子屏幕。
    *   **安全色**: 科技蓝 (`#41a5ea`) —— 代表“理智”和“表象”。
    *   **警告色**: 霓虹粉 (`#ff0055`) —— 代表“堕落”、“性欲”和“真相”。
    *   **字体**: `Roboto Mono` 或 `Consolas`，营造电子监控的数据感。

## 2. 界面布局规划 (HTML结构指导)

*   **顶部栏 (System Status)**：
    *   左侧：当前时间（数字时钟风格）、家庭安防状态（“SAFE” -> “WARNING” -> “CORRUPTED”）。
    *   右侧：**理智值 (Sanity)** 进度条（蓝色，越低画面越抖动）、**怀疑度 (Suspicion)** 进度条（红色，越高解锁更多调查选项）。

*   **左侧/主区域 (原地图区替代方案)**：
    *   **模块名称**: **全屋平面监控图 (Floor Plan Surveillance)**。
    *   **功能描述**: 使用 SVG 绘制家中户型图（客厅、主卧、女儿房、浴室）。
        *   每个房间是一个可点击区域。
        *   房间内显示当前的人物图标（丈夫、妻子、女儿）。
        *   **交互**: 点击房间可“前往查看”或“调取监控（如果有道具）”。
        *   **动态反馈**: 如果该房间内正在发生不可描述之事，房间区域会呈现红/粉色脉冲闪烁。

*   **右侧/状态区 (Target Profile)**：
    *   **目标监视卡片**: 分别展示 **妻子(苏婉)** 和 **女儿(李悦)** 的实时状态。
        *   **头像**: 带有故障艺术（Glitch）效果的头像。
        *   **数据**:
            *   **堕落度**: 也就是“母狗化进度”，使用粉色圆环图展示。
            *   **当前伪装**: 文字描述（如“正在做饭” -> “正在自慰”）。
        *   **威胁源 (王强)**: 仅在特定剧情触发时弹出的警告窗口。

*   **底部/互动区 (Command Center)**：
    *   **四大核心指令**: 四个占据底部宽度的矩形大按钮，对应“主角信息”表中的四个动态选项。
    *   **快捷工具栏**: “背包/证物袋”、“调查笔记”、“系统日志”。

## 3. 数据库表结构设计 (JSON Schema指导)

*   **表1：全局监控数据**
    *   核心字段：`地图标题` (当前区域), `当前时间`, `安防等级` (氛围描述), `轮次计数`, `外部区域列表`.
*   **表2：主要地点表 (房间)**
    *   核心字段：`地点名称` (房间名), `X坐标`, `Y坐标`, `宽度`, `高度`, `环境描述`.
*   **表3：家庭成员状态 (主角+NPC整合)**
    *   *注：为了简化前端逻辑，建议将所有人物状态整合或标准化，但为了兼容范例逻辑，我们保留“主角”和“重要人物”分离，但字段要对应。*
    *   **主角表**: `人物名称`, `理智值`, `怀疑度`, `当前位置`, `选项一`, `选项二`, `选项三`, `选项四`.
    *   **重要人物表**: `姓名`, `堕落度`, `当前状态` (伪装/真实), `所在地点`, `好感度` (此处挪用为服从度/羞耻度).
*   **表4：可疑痕迹表 (原地图元素)**
    *   核心字段：`痕迹名称` (如“陌生的烟头”), `类型` (物品/痕迹), `描述`, `所在房间`, `互动选项1-3`.
*   **表5：任务与推演**
    *   核心字段：`当前疑点`, `推演进度`, `真相碎片`.
*   **表6：小总结表** (标准结构)
*   **表7：大总结表** (标准结构)

## 4. 交互逻辑 (Prompt编写指导)

*   **触发更新的事件**：
    *   每当玩家执行“调查”、“对话”或“移动”指令后。
    *   每当**怀疑度**达到阈值（30%, 60%, 90%）时，UI 颜色风格发生突变（由蓝转红）。
*   **特殊视觉反馈**：
    *   **理智崩溃**: 当理智值 < 20% 时，前端 CSS 注入 `@keyframes shake` 动画，使整个界面持续轻微颤抖，模拟主角颤抖的手和崩溃的内心。
    *   **真相显现**: 当某个房间内“堕落度”过高的人物处于“暴露”状态时，该房间在地图上显示为深红色，并标记“高能反应”。
