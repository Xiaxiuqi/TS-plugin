# 备忘录表提示词

## 表格说明 (Note)

备忘录表用于保存可召回的关键细节，包括证据、原话、原文、感官细节、物理痕迹、阶段边界、多子项成果与约定内容。

- 列1: 标题 - 备忘录的简短标题。标题必须指向事件链、阶段节点、闭环结果、成果集合或约定本身，禁止用“抵达某地”“进入某房间”“进行一次对话”等小步骤作为标题。
- 列2: 分类 - 必须从以下五类中选择：秘密线索类、阶段节点类、大事件类、多子项成果类、约定承诺类。
  - 秘密线索类：记录独立、长期有效、不可更新正文的秘密线索。
  - 阶段节点类：记录长期事件中的一个可独立召回阶段。适用于案件、任务、关系线、调查线持续很久，但当前阶段已经形成局部结果的情况。
  - 大事件类：记录已经闭环或接近闭环的重大事件、成长节点、转折事件。也可用于不想扩展分类时承载阶段节点和多子项成果。
  - 多子项成果类：记录一组已完成或阶段性完成的并列成果，如研发一套10只盲盒、完成一批道具、设计多个角色方案等。
  - 约定承诺类：记录明确约定、承诺、契约、截止时间或触发条件。
- 列3: 归档状态 - 分为“未归档”、“已归档”、“已废除”。默认插入时为“未归档”。一旦设置为“已归档”或“已废除”，则本行所有字段均不可再更新。
- 列4: 记录日期 - 记录该备忘录的日期（游戏内时间，格式：YYYY-MM-DD 星期X HH:MM），与全局数据表时间格式一致。
- 列5: 记录内容 - 备忘录的具体内容。备忘录是纪要表的细节补充，不负责概括剧情；必须保留原始细节、原话、异常表现、因果依据、关键动作与关键对象。对于涉及多个子项的事件（如研发一套10只盲盒），应逐项列出关键属性（名称、设计、功能、数量等）。每条备忘录总token不得超过300token；若细节过多，不要强行概括，应按阶段节点拆成多条记录。
- 列6: 涉及对象 - 本次备忘录涉及到的角色、组织或特殊对象。
- 列7: 关键词 - 用于快速检索本条备忘录的标签。必须从正文或用户输入中提取能够**唯一指向本条备忘录的具体名词**（如特定物品名、技能名、约定代号、事件专属名称等）。严禁使用泛化的地点名（如“旧校舍”、“森林”）、泛化名词（如“徽章”、“任务”），避免因多事件共用同一地点或同类物品而触发错误召回。**每条备忘录的关键词最多5个，不要求写满；宁可少写，也不要为了凑数加入泛化词**，多个关键词用英文逗号分隔。

【细节保真原则】

严禁把具体细节改写成抽象概括。写入“记录内容”时必须遵守：

1. 原话保留：角色原话、残响原文、纸条文字、约定原句、提示语、特殊称呼必须尽量原样写入，不得改成概括词。例如“三号柜不要照灯”不能概括成“获得警告”或“出现关键提示”。
2. 感官细节保留：声音、气味、触感、视觉异常、身体反应、物理痕迹等必须写出具体表现，不得只写“发现异常”“逻辑错位”“获得线索”。
3. 行为细节保留：谁做了什么、用了什么能力/工具、对什么对象产生了什么结果，必须写清楚。
4. 反常点保留：若某事件的价值在于违和、矛盾、反常逻辑，必须写出“正常应当如何”和“实际表现如何”的差异。
5. 推理边界保留：可以写出基于细节得到的初步推论，但不能用推论替代细节。推论必须放在细节之后。
6. 不写空泛总结句：禁止只写“明确了核心违和点”“将目标转向具体空间”“完成线索锁定”等纪要式表达，除非前面已经写明具体证据。
7. 超长时拆分：如果保留细节会超过长度限制，应新增阶段节点或秘密线索记录，而不是删掉原话和细节来压缩。

【核心原则：阶段颗粒度与防增殖】

1. 备忘录表负责保存对后续有召回价值的具体细节。长期案件、长期任务、长期研发、长期关系线不必强行压缩成唯一一条备忘录。
2. 阶段内合并：同一阶段内的连续行动、对话、观察、推理、微小发现，必须合并进该阶段的未归档备忘录，不得每次推进都新增。
3. 阶段间拆分：当长期事件出现明确阶段边界时，允许新增一条“阶段节点类”备忘录。阶段边界包括：调查目标改变、行动场景与目标同时改变、获得独立线索组、能力/关系/阵营认知发生实质变化、阶段性成果完成、继续合并会导致旧行过度概括或丢失细节。
4. 小动作不记录：移动、抵达、进入房间、普通寒暄、无结论的零散观察、重复推理，不得单独新增。
5. 细节先于结果：每条备忘录必须先写可核查的具体细节，再写阶段性结果。不得只保留结果、删除细节。
6. 多子项成果保留：若事件内部包含多个独立且可并列的子成果（如一套10个设计各异的盲盒），应在对应备忘录中逐项列出每个子成果的具体信息。

【阶段节点判定规则】

同一长期事件中，只有出现以下情况之一，才允许新增阶段节点类备忘录：

1. 调查阶段切换：如“接触委托方”→“现场调查”→“线索验证”→“锁定对象”→“收尾处理”。
2. 场景阶段切换且目标改变：不是单纯抵达新地点，而是抵达后开始新的调查目标或行动目标。
3. 关键线索独立成组：获得的信息足够完整，可独立支持后续推理，且包含原话、感官表现、物理证据或明确对象，而不是单句碎片。
4. 能力、关系、阵营认知发生实质变化：如确认能力边界、建立合作关系、NPC立场改变。
5. 阶段性成果完成：如完成一组道具设计、破解一个机关、确认一个嫌疑方向、签订一个约定。
6. 旧备忘录接近长度上限：若继续合并会导致记录内容超过限制、被迫概括、丢失关键细节，允许新增同事件链的下一阶段备忘录。

禁止新增阶段节点类备忘录的情况：

1. 单纯移动、抵达、进入、离开。
2. 普通寒暄、情绪反应、无长期影响的对话。
3. 对同一线索的短暂重复解释。
4. 未产生阶段性结论的零散观察。
5. 同一场景内连续出现的同类小发现。
6. 只换了陪同角色但目标未变化。
7. 只是上一条备忘录的同义补充。

【同事件链标题规则】

同一长期事件拆分为多个阶段备忘录时，标题必须使用统一前缀。

格式：事件名：阶段节点

示例：

- `霜灯温室：保管员委托与钥匙交接`
- `霜灯温室：夜间巡查与露水异常`
- `霜灯温室：叶脉留言线索`
- `霜灯温室：温控破坏对象锁定`

禁止使用无事件前缀的小标题：

- `抵达现场`
- `隔间异常`
- `再次推理`
- `与NPC交谈`

【长期事件关键词规则】

若备忘录属于某个长期事件的阶段节点，关键词建议采用：事件链关键词,阶段唯一关键词,关键物品/能力/对象关键词。

- 第一个关键词用于聚合同一长期事件。
- 第二个关键词用于定位本阶段的独特细节。
- 第三个至第五个关键词可用于记录本阶段的关键物品、能力、对象、代号或子成果名。
- 禁止使用泛化词凑数；宁可少于5个，也不要加入“线索”“任务”“现场”等泛化词。
- 禁止每次更新都随意更换事件链关键词。

示例：

- 推荐：`霜灯温室,银叶钥匙,月眠种`
- 推荐：`霜灯温室,叶脉留言,第三层温室`
- 推荐：`霜灯温室,霜斑温控阀,露水倒流`
- 不推荐：`现场调查,能力测试`
- 不推荐：`温室,线索`

【分类与更新规则】

- 秘密线索类：不可更新记录内容、涉及对象、关键词。用于记录“获得线索”的事实本身，但线索的具体内容应在“记录内容”中完整写出，尤其要保留原话、原文、感官表现、发现方式和反常点。若某线索只是当前阶段节点中的过程性发现，但细节很关键，也允许单独新增秘密线索类；阶段节点中只需引用该线索标题或关键词。
- 阶段节点类：可更新。用于记录长期事件中一个可独立召回的阶段。阶段内有新细节时更新本行；进入新阶段时可新增下一条阶段节点。更新时不得把关键原话、感官细节、物理证据压缩成抽象结论。
- 大事件类：可更新。用于记录重大闭环事件、转折事件，或在不使用扩展分类时承载阶段节点、多子项成果。大事件类也必须保留关键细节，不得写成纪要。
- 多子项成果类：可更新。用于记录多项并列成果。更新时必须保留已确认子项，不得因压缩而丢失子项名称和关键属性。
- 约定承诺类：不可更新记录内容、涉及对象、关键词。用于记录约定的具体内容、参与方、截止时间、触发条件等。

【归档状态规则】

- 未归档：可更新（根据分类规则）记录内容、涉及对象、关键词、归档状态。
- 已归档：表示事件完成、线索已用、约定已履行、阶段已收束，禁止任何更新操作。
- 已废除：表示记录错误、线索被证伪、约定被取消、重复流水账已被合并，禁止任何更新操作。

【多子项事件规则】

对于研发一套10只盲盒、制作一批道具、完成多个并列角色设计等多子项事件：

1. 优先归入“多子项成果类”；若数据库不启用扩展分类，则归入“大事件类”。
2. 必须等到整套成果完成，或取得明确阶段性最终结果后再写入。
3. 记录内容中必须逐项列出每个子项的名称、设计、功能、数量或关键差异。
4. 关键词最多5个，不要求写满；应优先选择系列总名、项目代号或成果总名，例如“幻彩盲盒,十只盲盒交付”。只有子项名称本身足够独特且对召回有必要时，才可补充为第3到第5个关键词。
5. 不要把所有子项名称都塞进关键词；子项名称应放在记录内容中，以保证可识别性，同时避免关键词爆炸。

【检查清单】

1. 当前内容是否只是小动作、小对话、小发现？若是，不写入。
2. 当前内容是否属于已有阶段的延续？若是，更新已有阶段节点类、大事件类或多子项成果类记录。
3. 当前内容是否形成新阶段节点？若是，允许新增阶段节点类记录。
4. 当前内容是否形成完整闭环大事件？若是，允许新增或更新大事件类记录。
5. 当前内容是否是多子项成果？若是，是否逐项列出每个子项的名称、设计、功能、数量等关键信息？
6. 分类是否为允许分类之一？
7. 归档状态是否为“未归档”？只有“未归档”时才允许更新。
8. 关键词是否采用具体名词，且数量最多5个、不强行写满？
9. 若属于长期事件，标题是否使用“事件名：阶段节点”格式？关键词是否使用“事件链关键词,阶段唯一关键词,关键物品/能力/对象关键词”格式？
10. 如果继续合并会导致过度概括或丢失细节，是否应新增阶段节点而不是强行更新旧行？
11. 是否保留了原话、原文、感官细节、物理痕迹、能力/工具名称、反常点？若没有，禁止写入。
12. 是否出现了“线索锁定、核心违和点、逻辑闭环、目标转向、阶段推进”等只有结论没有细节的纪要式表述？若有，必须补回具体证据。

## 初始化触发 (Init)

游戏初始化时，如有已存在的重要备忘录则插入条目，分类必须明确，归档状态默认为“未归档”。初始化并不意味着可以导入流水账；仍必须按阶段颗粒度整理。

允许初始化插入的内容：

1. 已存在且对主角状态有长期影响的重大事件。
2. 已存在且独立有效的秘密线索。
3. 已存在且尚未完成或仍有约束力的约定承诺。
4. 已完成或已取得阶段性最终结果的多子项成果事件。
5. 长期事件中已经形成清晰边界的阶段节点。

禁止初始化插入的内容：

1. 旅途移动、抵达、普通寒暄、准备行动。
2. 同一阶段中的每个小步骤。
3. 已能合并为一个阶段节点的多条过程性记录。

-- 示例1：秘密线索类（单条）
INSERT INTO memo (row_id, title, category, archive_status, record_date, content, involved_objects, keywords)

## 新增触发 (Insert)

只有当一个完整的事件闭环结束、取得阶段性的最终结果、形成新阶段节点、获得独立秘密线索、形成明确约定，或完成多子项成果集合时才可添加。

新增前必须先查找是否已有可合并的未归档记录：
SELECT row_id, title, category, archive_status, record_date, content, involved_objects, keywords
FROM memo
WHERE archive_status = '未归档'
  AND category IN ('阶段节点类', '大事件类', '多子项成果类')
  AND (
    title LIKE '%' || '霜灯温室' || '%'
    OR content LIKE '%' || '银叶钥匙' || '%'
    OR keywords LIKE '%' || '霜灯温室' || '%'
  );

上方查重 SQL 中，`'霜灯温室'` 和 `'银叶钥匙'` 是示例检索值，实际填表时必须替换为当前事件的真实事件链关键词、阶段唯一关键词或成果总名；保留外层单引号，不要写成未加引号的中文裸词。若检索值本身包含英文单引号，必须先按 SQLite 字符串规则转义为两个单引号。示例：`'角色''代号'`。

允许新增的情况：

1. 一个完整事件闭环结束，并产生决定性结果。
2. 一个长期事件取得可独立总结的阶段性最终结果。
3. 同一长期事件进入新的阶段节点，且旧阶段继续合并会导致概括过度或细节丢失。
4. 获得一条独立、不可更新的秘密线索，且该线索不是现有阶段节点中已经记录的过程性子线索。
5. 形成一项明确约定、承诺、契约，且包含参与方、内容、截止时间或触发条件。
6. 完成一个多子项成果集合，且每个子项已经形成可列举的名称、设计、功能或数量。

禁止新增的情况：

1. 路上移动、抵达地点、进入房间、准备调查。
2. 对同一线索的重复解释、几种推演、逻辑自白、阶段性思考。
3. 同一阶段中连续出现的微小发现。
4. 已被现有未归档阶段节点或大事件覆盖的秘密线索。
5. 仅改变场景、位置、陪同角色，但目标未变的行动。
6. 多子项成果尚未完成，仅完成其中一两个子项，且还没有形成阶段性最终成果。

SQL示例：同初始化插入语句。

-- 示例5：同一长期事件拆分为多个阶段节点，而不是每次推进都新增
INSERT INTO memo (row_id, title, category, archive_status, record_date, content, involved_objects, keywords)
VALUES ((SELECT COALESCE(MAX(row_id), 0) + 1 FROM memo), '霜灯温室：夜间巡查与露水异常', '阶段节点类', '未归档', '2024-06-04 星期二 21:10', '主角与植物学徒周栖进入北侧侧门。第三层温室门缝持续滴下温热露水，但门把手结着硬霜；周栖用温控笔检测到门内为28℃、门外为6℃。主角听见玻璃顶上传来三次空钟声，露水随钟声逆着门缝向上爬。温热水滴、结霜把手、反向流动三者互相矛盾，初步确认温控阀或空间湿度记录被人为改动。', '主角, 周栖', '霜灯温室,霜斑温控阀');

## 更新触发 (Update)

根据分类和归档状态决定是否可更新：若归档状态为“已归档”或“已废除”，则禁止任何更新。若归档状态为“未归档”：秘密线索类、约定承诺类仅可更新“归档状态”；阶段节点类、大事件类、多子项成果类可更新“记录日期”、“记录内容”、“涉及对象”、“关键词”、“归档状态”。任何情况下均不可更新“分类”。

更新规则：

1. 秘密线索类：不可更新“标题”、“分类”、“记录日期”、“记录内容”、“涉及对象”、“关键词”，只允许将“归档状态”从“未归档”改为“已归档”或“已废除”。
2. 约定承诺类：不可更新“标题”、“分类”、“记录日期”、“记录内容”、“涉及对象”、“关键词”，只允许将“归档状态”从“未归档”改为“已归档”或“已废除”。
3. 阶段节点类：若“归档状态”为“未归档”，可更新“记录日期”、“记录内容”、“涉及对象”、“关键词”、“归档状态”。更新时只合并同一阶段内的新细节；若已经进入新阶段，应新增下一条阶段节点，而不是继续压缩旧行。
4. 大事件类：若“归档状态”为“未归档”，可更新“记录日期”、“记录内容”、“涉及对象”、“关键词”、“归档状态”。适合闭环事件或重大转折的持续补充。
5. 多子项成果类：若“归档状态”为“未归档”，可更新“记录日期”、“记录内容”、“涉及对象”、“关键词”、“归档状态”。更新时必须保留所有已确认子项名称和关键属性。
6. 任何分类都禁止更新“分类”。
7. “已归档”或“已废除”的行禁止任何更新。

阶段节点类、大事件类、多子项成果类更新时必须遵守：

1. 读取旧的记录内容。
2. 保留旧内容中的长期有效事实、关键线索、已确认子项。
3. 在不丢失原话、感官表现、物理痕迹、工具名称和反常点的前提下，把同阶段新旧信息整理为一条细节记录。
4. 删除路上移动、普通对话、重复推理等低价值过程。
5. 保留决定性结果和对后续有影响的细节。
6. 若是多子项成果，逐项保留每个子项的名称、设计、功能、数量，不得因压缩而丢失已确认子项。
7. 关键词最多保留5个，不要求写满；保留项必须指向整条事件链、阶段独特细节、关键物品、关键能力、关键对象或成果总名。

-- 示例6：更新同一阶段节点
UPDATE memo
SET
  record_date = '2024-06-04 星期二 10:05',
  content = '主角在晨检时发现三号玻璃温室内侧凝着蓝白霜斑，霜纹呈向内收拢的叶脉形。叶澜说明昨夜总温控没有报警，但珍稀种子柜少了一格“月眠种”。叶澜交给主角银叶钥匙和温室巡检册，要求主角只在日落后打开北侧侧门；补充检查时，主角发现巡检册昨夜23:10一栏被水浸开，只剩“勿照灯”三字。',
  involved_objects = '主角, 叶澜',
  keywords = '霜灯温室,银叶钥匙'
WHERE title = '霜灯温室：保管员委托与钥匙交接'
  AND category = '阶段节点类'
  AND archive_status = '未归档';

-- 示例7：归档秘密线索
UPDATE memo
SET archive_status = '已归档'
WHERE title = '初始线索'
  AND category = '秘密线索类'
  AND archive_status = '未归档';

## 删除触发 (Delete)

禁止删除。

不得使用 `DELETE FROM memo` 删除正式剧情记录。若某条记录错误、重复、被证伪或已经被合并进更合适的阶段节点，应将“归档状态”更新为“已废除”或“已归档”，而不是物理删除。

-- 示例8：废除重复流水账
UPDATE memo
SET archive_status = '已废除'
WHERE title = '抵达海岬灯塔调查现场'
  AND archive_status = '未归档';

## DDL 定义 (SQLite 模式)

CREATE TABLE 语句:
CREATE TABLE memo ( -- 备忘录表
  row_id INTEGER PRIMARY KEY, -- 行号
  title TEXT NOT NULL, -- 标题
  category TEXT NOT NULL CHECK(category IN ('秘密线索类', '阶段节点类', '大事件类', '多子项成果类', '约定承诺类')), -- 分类
  archive_status TEXT NOT NULL DEFAULT '未归档' CHECK(archive_status IN ('未归档', '已归档', '已废除')), -- 归档状态
  record_date TEXT NOT NULL, -- 记录日期
  content TEXT NOT NULL CHECK(LENGTH(content) <= 600), -- 记录内容
  involved_objects TEXT, -- 涉及对象
  keywords TEXT -- 关键词
);

可选索引：
CREATE INDEX IF NOT EXISTS idx_memo_status_category ON memo (archive_status, category);
CREATE INDEX IF NOT EXISTS idx_memo_title ON memo (title);
CREATE INDEX IF NOT EXISTS idx_memo_keywords ON memo (keywords);

可选触发器：禁止更新已归档或已废除行。
CREATE TRIGGER IF NOT EXISTS trg_memo_no_update_archived
BEFORE UPDATE ON memo
FOR EACH ROW
WHEN OLD.archive_status IN ('已归档', '已废除')
BEGIN
  SELECT RAISE(ABORT, '已归档或已废除的 memo 行禁止更新');
END;

可选触发器：禁止修改分类。
CREATE TRIGGER IF NOT EXISTS trg_memo_no_category_change
BEFORE UPDATE OF category ON memo
FOR EACH ROW
WHEN OLD.category <> NEW.category
BEGIN
  SELECT RAISE(ABORT, 'memo 分类禁止更新');
END;
