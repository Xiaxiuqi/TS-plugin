# 剧情推进预设提示词

## 这个文件负责什么

当组件规划已经确认“需要剧情推进预设”时，使用本文件。

本文件只负责：

- `frontend/剧情推进.json`

如果还没有确认组件结构，先回到：
- `prompts/7-开场白状态栏讨论.md`

如果要做的是推进美化 HTML 与推进正则配置，改去：
- `prompts/7-4-剧情推进前端.md`

---

## ⚠️⚠️⚠️ 最高优先级规则：开始前必须先阅读本文件与范例 ⚠️⚠️⚠️

在讨论或生成剧情推进预设之前，必须先阅读：

1. `prompts/7-3-剧情推进预设.md`
2. `examples/full-card/轮回禁则/frontend/剧情推进.json`

### 阅读目标

- 学习剧情推进预设的固定结构
- 学习 recall 相关字段的语义
- 学习 `extractTags` 应如何定义
- 学习“预设本体先正确，再谈展示层”的优先级

### 禁止行为

1. ❌ 禁止不读范例就直接写 `剧情推进.json`
2. ❌ 禁止让推进美化层反向定义 recall 规则
3. ❌ 禁止把预设拆成另一套自创 schema
4. ❌ 禁止没有正确对象反馈时仍然强行填充 `<recall>`

---

## 阶段目标

与用户确认并完成：

1. 是否真的需要剧情推进预设
2. 预设要承担哪些用途
3. recall 规则如何定义
4. `extractTags` 如何规划
5. 最终形成固定格式的 `frontend/剧情推进.json`

---

## 什么情况下需要剧情推进预设

只要用户有以下任一需求，就应考虑启用：

- 正文前额外推进
- 最终注入替换
- 额外 DM 判定
- 规则判定辅助
- 需要结构化推进结果供前端美化消费

如果用户只要普通世界书和普通状态栏，且没有上述链路，可以不做。

---

## 核心优先级：预设本体高于展示层

### 强制规则

- 先把 `剧情推进.json` 做对
- 再去做 `推进美化.html` 与 `推进正则配置.json`
- 展示层只能消费已经正确产出的标签
- 展示层不能倒逼预设层改语义

### 绝对禁止

- ❌ 为了让美化界面好看，回头改 recall 逻辑
- ❌ 在预设没有正确定义前，先做推进前端
- ❌ 用推进美化层掩盖预设本体错误

---

## recall 规则是本文件的最高重点

### 必须保留 recall 相关字段

例如：
- `recallCount`
- `extractTags`
- 以及范例结构中的相关召回定义位

### `zhaohui` 的语义

`zhaohui` 必须视为：

- **召回数量的数字占位符**
- 不是标签名
- 不是文本装饰词
- 不能写成“zhaohui条”这种易混淆表述

### 推荐固定表述

```text
召回zhaohui数量的总结条目；若不足则按最大可召回条数处理
```

### recall 的固定语义

1. 只从正确对象反馈中提取编码
2. 背景设定、前文剧情、上轮规划只用于理解局势
3. 这些上下文不能直接作为 recall 编码来源
4. 如果没有正确对象反馈，`<recall>` 必须明确留空
5. 即使给了数字，也不能为了凑数量去补造内容

---

## `extractTags` 规划规则

### 作用

`extractTags` 是后续推进前端展示层要消费的标签清单。

### 要求

1. 必须显式列出
2. 必须和预设实际输出一致
3. 不要写一堆不会实际产出的标签
4. 不要把展示层想要的标签硬塞给预设层

### 原则

- 先想清楚预设真正会输出什么
- 再把这些东西列为 `extractTags`
- 后续推进前端只能围绕这些标签去美化

---

## 先向用户确认什么

### 预设确认模板

```text
在制作剧情推进预设前，我先确认以下内容：

【用途】
- 是否需要正文前额外推进？
- 是否需要最终注入替换？
- 是否需要额外 DM 判定？
- 是否需要规则判定辅助？

【召回规则】
- 是否需要 recall？
- recall 应召回哪些总结条目？
- 没有正确对象反馈时，是否明确保持 recall 为空？（正确答案通常是“是”）

【输出标签】
- 预设最终需要输出哪些结构化标签供后续展示消费？

确认后，我会基于范例格式起草 `剧情推进.json`。
```

## 文件格式要求

### 强制规则

1. 必须保持与范例兼容的固定 JSON 结构
2. 最终注入模板与推进逻辑必须在同一个预设里
3. 不要拆出另一套 schema
4. 需要保存到：`frontend/剧情推进.json`

### 禁止行为

- ❌ 自创另一种字段组织方式
- ❌ 为了省事删掉 recall 相关字段
- ❌ 把展示相关逻辑直接混成 HTML

## 与开场白、推进前端的关系

### 与开场白前端的关系

如果开场白需要在开始按钮中一并注入预设：
- 预设必须先定稿
- 再去做 `opening-regex.html`

### 与推进前端的关系

推进前端必须建立在预设已经定稿的基础上：
- 先定 `extractTags`
- 先定 recall 规则
- 再做美化与 cleanup 配置

---

## 草稿展示时必须写清楚的内容

展示预设草稿时，必须同时告诉用户：

1. 本预设承担的功能范围
2. recall 的来源与空值规则
3. `extractTags` 清单
4. 为什么这些标签足够支撑后续推进前端

### 展示模板

```text
剧情推进预设草稿如下：

【功能范围】
- [正文前推进 / 最终注入 / DM 判定 / 规则辅助]

【recall 规则】
- 来源：正确对象反馈
- 数量占位：zhaohui
- 无正确对象反馈时：recall 留空

【extractTags】
- 标签A
- 标签B
- 标签C

【说明】
- 这些标签将供后续推进前端展示使用，但不会反向修改预设规则
```
