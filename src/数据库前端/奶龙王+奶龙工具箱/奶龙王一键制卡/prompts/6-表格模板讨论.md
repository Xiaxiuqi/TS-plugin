## 表格系统说明

表格系统是角色卡的核心组件，用于：
- **纪要表**：记录剧情进展，自动生成世界书条目
- **重要角色表**：记录登场角色，自动生成角色索引
- **自定义表格**：根据需求创建属性表、物品表等

### 表格与世界书的职责边界（强制规则）

- 数据表负责记录、约束和更新变量；世界书负责读取变量、判断条件、提供表现与叙事指令。
- 所有变量的涨跌规则、增减幅度、累计口径、锁定机制、恢复机制，必须只写在对应数据表的 `## 说明` 中。
- 世界书条目只允许知道数据表的条件判断规则，不允许定义变量的具体数值涨跌幅。
- 堕落事件、剧情事件、状态切换事件只负责触发条件、触发事件、解锁结果、阶段切换、能力变化和叙事后果，不写具体变量加减数字。
- 设计新表时，必须先判断哪些规则属于“变量更新规则”，并优先写入表说明；不要把变量更新规则分散写进世界书条目。
- 写变量规则时，优先写清“这个字段应该怎么写”以及“这个字段应该怎么变”：字段职责、字段格式、来源归属、计量口径、总累计还是当前值、变化规则、必要边界。
- 变量变化规则必须是可执行、可判定的。AI看完后应当知道在什么情况下记小幅变化、明显变化、重大变化；不能只写“由某些因素推动”“不会无故变化”这类抽象空话。
- 不要优先堆砌大量禁止项、补丁项或 case-by-case 例外；规则应当先给出正向写法，再用少量必要约束防止跑偏。
- 规则负责给边界，不负责替 AI 穷举所有情形；应保留合理发挥空间。
- 重写数据表时，必须同时检查 `## 说明` 和 `## 更新规则` 是否职责混乱，不能只改一半。
- `## 说明` 负责字段职责、格式、来源、计量口径、变量更新规则与必要边界；如果某张表负责某个变量，就必须在 `## 说明` 中写清这个变量怎么变。
- `## 更新规则` 只允许保留最小操作规则：初始化怎么写、更新时改哪里、是否允许插入、是否允许删除。
- `## 更新规则` 不得重复 `## 说明` 已经承担的内容；凡是字段定义、变量逻辑、来源口径、解释性约束，应留在 `## 说明`，不应再次塞进 `## 更新规则`。

### 表格与世界书的关联

表格可以通过 `exportConfig` 自动生成世界书条目：
- `enabled: true` - 启用条目生成
- `splitByRow: true` - 每行生成一个独立条目
- `entryType: "keyword"` - 关键词触发
- `keywords: "列名"` - 指定触发关键词所在的列

### 表格分组与并发填表逻辑（新增）

数据库插件会根据 `updateConfig.contextDepth` 和 `updateConfig.batchSize` 的**组合值**给表格分组：

- **同组规则**：两个字段都相同 → 这些表会被视为同一组，进入同一批次填表
- **拆组规则**：只要任意一个字段不同 → 会被拆到不同组，可并发填表
- **适用场景**：
  - 信息结构相近、依赖同一段上下文、复杂度接近的表，可以放在同一组
  - 字段很多、文本很长、推理负担重的表，可以考虑拆组并发
  - **但如果某张表承担“规则说明 / 特殊规则 / 计算依据 / 校验依据”的职责，它不能脱离依赖它的业务表单独拆组**
- **目标**：通过有意识地调整这两个参数，把真正的高压表拆开，把轻量表归并，同时确保“规则表”和“使用该规则的表”在同批更新时能彼此看见
- **推荐约束**：若无明确理由，`contextDepth` 与 `batchSize` 建议保持相同数值，优先使用 `4/4`、`5/5`、`8/8` 这类同值组合，便于统一维护和识别分组
- **补充说明**：如果参数写成 `-1`，表示该表使用全局设置，而不是在当前表上单独声明一个新的值
- **分组注意**：如果要精确控制某张表的分组，请不要把对应的 `contextDepth` / `batchSize` 留成 `-1`；若保留 `-1`，则该表的实际分组效果取决于全局设置最终生效后的值

#### 制表时必须执行的分组步骤

1. 先列出本次要做的所有表
2. 判断每张表属于：核心状态表 / 规则支撑表 / 长文本表 / 战斗专用表 / 轻量记录表
3. 先找出哪些业务表需要依赖规则表、校验表或特殊规则表
4. 将“规则表 + 依赖该规则的业务表”强制放入同一组，再考虑其余表如何拆组
5. 为每一类表分配一组 `contextDepth + batchSize` 组合，默认优先使用同值配对
6. 同组表必须使用完全相同的两个值，且优先保持 `contextDepth = batchSize`
7. 不同组至少有一个值必须不同，避免插件误并组
8. 在向用户展示草稿时，**必须额外展示分组方案**，说明每组包含哪些表、为什么这样分

#### 推荐分组原则

- **核心状态联动表**：通常可共组，建议使用相同的中等 `contextDepth`
- **纪要 / 长文本总结表**：可优先单独成组，但前提是它不承担其他表的规则支撑职责
- **特殊规则表 / 系统规则表 / 校验表**：必须与使用这些规则的业务表同组，不能为了减压而盲目拆开
- **战斗表**：若依赖战斗规则、数值规则或结算规则，应与对应规则表同组
- **纯记录型轻量表**：可以适度合并到同组，降低总批次数
- **默认上限**：一般不要拆成超过 3 组，除非用户明确要求做更细的特殊分组

## 表格设计原则

### 1. 行列结构定义规范（强制规则）

**所有表格（除纪要表、重要角色表外）必须在 `## 说明` 中严格定义行列结构：**

#### 格式要求

```
【表格结构】
- 行0: 表头 - 列名说明
- 行1~N: 数据 - 每行代表什么
- 列0: 字段名 - 字段说明（关键词触发列）
- 列1: 字段名 - 字段说明
- 列2~N: 字段名 - 字段说明
```

#### 不同类型表格的结构定义示例

**横排表（一行一条记录）**：
```
【表格结构】
- 行0: 表头 - 列名（角色1、角色2、关系、亲密值、特殊状态）
- 行1~N: 数据 - 每行代表一组角色关系
- 列0: 角色1 - 关系的一方
- 列1: 角色2 - 关系的另一方
- 列2: 关系 - 两人的关系类型
- 列3: 亲密值 - 关系亲密程度的数值
- 列4: 特殊状态 - 关系的特殊状态标记
```

**竖排键值对表（字段-值结构）**：
```
【表格结构】
- 行0: 表头 - 列名（字段、当前值）
- 行1: 字段名1 - 该字段的说明
- 行2: 字段名2 - 该字段的说明
- 行3: 字段名3 - 该字段的说明
- 列0: 字段 - 属性名称（关键词触发列）
- 列1: 当前值 - 该属性的当前值
```

**行列颠倒表（属性在行，角色在列）**：
```
【表格结构】
- 行0: 表头 - 第一列为「属性」，其余列为角色名
- 行1~10: 数据 - 每行代表一个状态属性
- 列0: 属性 - 各项状态属性的名称（关键词触发列）
- 列1: 角色A - 角色A的状态数据
- 列2: 角色B - 角色B的状态数据
- 列3~N: 其他角色 - 其他角色的状态数据
```

#### 定义规则

1. **必须标注关键词触发列**：用 `（关键词触发列）` 标注哪一列用于关键词触发
2. **行定义必须说明行数范围**：`行0` 表示表头，`行1~N` 表示数据行
3. **列定义必须说明列职责**：每个列都要写清楚存什么数据
4. **行列颠倒表要明确说明**：如果表是属性在行、对象在列，必须在结构说明中明确标注

#### 适用范围

- ✅ **必须定义**：当前状态表、关系状态表、角色状态总表、身体变化总表、事件表等自定义表格
- ❌ **无需定义**：纪要表、重要角色表（已有固定格式说明）

### 2. 其他设计原则

1. **简洁性**：表格字段不宜过多，保持5-7个核心字段
2. **实用性**：每个字段都应该有明确的用途
3. **可扩展性**：预留备注字段用于记录额外信息
4. **必需表格优先**：纪要表和重要角色表必须存在
5. **exportConfig配置**：
   - 需要生成世界书条目的表格设置 `enabled: true`
   - 不需要的设置 `enabled: false`（仅供前端显示）
