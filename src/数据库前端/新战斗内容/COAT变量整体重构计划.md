# COAT 变量整体重构计划

## 1. 联想扩展（Associated Thoughts）

### 技术上下文联想

- 当前项目使用 MVU 变量框架，变量结构通过 `registerMvuSchema(Schema)` 注册。
- 变量数据最终存储于消息楼层变量的 `stat_data` 中。
- 现有前端读取路径集中在：
  - `stat_data.user.咒力`
  - `stat_data.user.战斗面板`
  - `stat_data.user.基础肉体`
  - `stat_data.user.生得术式`
  - `stat_data.user.扩展术式`
  - `stat_data.user.特殊体质`
- 新战斗框架要求将血量、防御、术式攻击、物理攻击、咒力回复、反转术式、术式熔断、大脑重置、术式潜力分级统一纳入变量体系。
- 变量结构需要继续保持 MVU 稳定性：
  - 空数组使用 `[]`
  - 空对象使用 `{}`
  - 不再使用 `待初始化`
  - 动态集合使用 `z.record`
  - 派生字段尽量在 schema transform 中自动结算

### 功能需求联想

- 核心功能：
  - 重构战斗属性字段
  - 根据新版公式自动结算派生值
  - 支持特殊体质补正
  - 支持反转术式治疗与熔断修复记录
  - 支持术式潜力分级与战力评级映射
- 辅助功能：
  - 保留前端已有读取字段，避免状态栏完全失效
  - 让变量更新规则明确哪些字段允许 AI 写入、哪些字段由结构自动计算
  - 初始化文件同步改为新版结构
- 错误处理：
  - 数值钳制
  - 缺省对象兜底
  - 对异常负数和超过上限字段做约束

### 数据流设计

1. AI/MVU 写入基础字段：
   - 咒力总量
   - 当前咒力
   - 咒力操纵精度
   - 基础肉体值
   - 武艺值
   - 生得术式潜力值
   - 生得术式精通等级
   - 特殊体质
   - 反转术式熟练度
   - 熔断状态与大脑重置次数
2. Schema transform 自动计算：
   - 术式强度_BPB
   - 总肉体值_BPA
   - 防御
   - 最大生命值
   - 总BP
   - 战力评级
   - 咒术攻击
   - 物理攻击
   - 咒力回复量_每轮
   - 常规治疗回复量
   - 常规治疗消耗
   - 熔断修复消耗
   - 术式强度惩罚系数
   - 咒力回复惩罚系数
3. 前端继续读取兼容字段：
   - `战斗面板.最大生命值`
   - `战斗面板.当前血量`
   - `战斗面板.防御`
   - `战斗面板.总BP`
   - `战斗面板.总肉体值_BPA`
   - `战斗面板.术式强度_BPB`

### 边界情况

- 天与咒缚（纯肉体）：咒力总量强制视为 0，肉体值 ×2，物理补正 ×2。
- 天与咒缚（咒力异化）：有效咒力总量 ×3，生命上限 ×0.75，伤害提升 20%，术式潜力 ×0.3 计入肉体值。
- 六眼：咒力消耗 ×1/1000，咒力输出程度恒定 100%，精度允许突破 2.0。
- 术式熔断：术式强度惩罚至 0 或按大脑重置惩罚系数结算。
- 大脑重置 3 次后开始惩罚，4 次重罚，5 次近乎归零。
- 行囊数量 <= 0 仍自动删除。

## 2. 候选方案（Candidate Actions）

### 方案 1：兼容式整体重构（推荐）

├── 实现路径:
│   - 在 `user` 下重构 `咒力`、`战斗面板`、`基础肉体`、`生得术式`
│   - 新增 `战斗计算` 模块承载咒术攻击、物理攻击、回复量、消耗、惩罚系数
│   - 新增 `反转术式` 模块承载治疗与熔断修复能力
│   - 新增 `术式熔断` 模块承载熔断状态、大脑重置次数和惩罚
│   - 保留旧前端依赖字段名 `总BP`、`总肉体值_BPA`、`术式强度_BPB`
│   - 在 schema transform 中自动计算派生战斗值
├── 技术栈选择:
│   - Zod 4 schema
│   - Lodash clamp/pickBy
│   - MVU registerMvuSchema
├── 开发工作量预估:
│   - 核心代码量：260~360 行
│   - 规则文档：350~500 行
│   - 初始化：120~180 行
│   - 集成调试：中等
└── 风险点:
    - 前端部分展示仍是旧命名，但兼容字段可缓解
    - 公式较多，transform 需要避免顺序错误
    - 特殊体质识别依赖字符串匹配，需要规则中统一命名

### 方案 2：完全领域模型重构

├── 实现路径:
│   - 完全移除旧 `战斗面板`
│   - 改为 `基础属性`、`派生属性`、`战斗状态`、`输出结算` 四大模块
│   - 前端也同步改为新字段
├── 技术栈选择:
│   - Zod 4 schema
│   - 前端同步重写
├── 开发工作量预估:
│   - 核心代码量：400~600 行
│   - 前端改动：大量
│   - 集成调试：较高
└── 风险点:
    - 会打断现有前端读取
    - 一次性改动过大
    - 排查问题成本高

### 方案 3：规则文档优先，schema 只加少量字段

├── 实现路径:
│   - 变量结构只新增少数字段
│   - 大部分公式交给变量更新规则约束 AI 手算
├── 技术栈选择:
│   - 文档规则为主
│   - schema 校验为辅
├── 开发工作量预估:
│   - 核心代码量：80~120 行
│   - 规则文档：较多
│   - 集成调试：较低
└── 风险点:
    - AI 手算容易出错
    - 前端与变量表现不稳定
    - 无法保证公式一致性

## 5. 落盘（Commit）

## 最终技术方案

### 架构设计

- 核心模块：`咒力`
  - 管理咒力总量、当前咒力、咒力操纵精度、有效咒力总量、每轮回复量。
- 核心模块：`基础肉体`
  - 管理基础肉体值、武艺值、武艺阶段、输出比例。
- 核心模块：`生得术式`
  - 管理术式名称、潜力等级、潜力值、精通等级、阶段、基础强度。
- 核心模块：`反转术式`
  - 管理是否掌握治疗、熟练等级、外放能力、常规回复、治疗消耗。
- 核心模块：`术式熔断`
  - 管理熔断状态、大脑重置次数、回复惩罚、术式强度惩罚、熔断修复消耗。
- 核心模块：`战斗面板`
  - 保留前端兼容字段，并新增咒术攻击、物理攻击、当前/最大生命值、防御、总BP、战力等级。
- 数据流：
  - 基础输入字段 → schema transform 自动派生 → MVU 存储到 `stat_data` → 前端读取兼容字段展示。
- 接口定义：
  - 输入：MVU 变量更新命令写入基础字段。
  - 输出：schema parse/transform 后的稳定 `stat_data.user`。

### 实现步骤

1. 重构变量结构：
   - 增加辅助函数。
   - 重构 `咒力`、`战斗面板`、`基础肉体`、`生得术式`。
   - 新增 `反转术式`、`术式熔断`、`战斗计算`。
2. 重构变量更新规则：
   - 明确基础字段可更新。
   - 明确派生字段禁止直接更新。
   - 写入特殊体质时使用统一名称。
3. 重构变量初始化：
   - 按新版结构补齐所有核心字段。
   - 初始化派生字段为可被 transform 覆盖的安全值。
4. 一致性复核：
   - 检查 schema、规则、初始化字段是否同名。
   - 检查前端兼容字段是否保留。
5. 输出交付：
   - 修改 `新变量结构`
   - 修改 `变量更新规则.txt`
   - 修改 `变量初始化`

### 关键代码片段（伪代码）

```ts
const 有效咒力总量 = hasTrait('天与咒缚（纯肉体）') ? 0
  : hasTrait('天与咒缚（咒力异化）') ? 咒力总量 * 3
  : 咒力总量;

const 术式强度 = 熔断中
  ? 基础术式强度 * 术式强度惩罚系数
  : 基础术式强度;

const 总肉体值 = (基础肉体值 + 有效咒力总量 * 0.1 * 咒力操纵精度 + 武艺值 + 异化术式补正) * 肉体补正;

const 防御 = 有效咒力总量 * 0.2 * 咒力操纵精度 * 武艺防御系数;

const 最大生命值 = ((有效咒力总量 * 咒力操纵精度) + 总肉体值 + 术式强度 + 反转术式补正) * 生命补正;

const 咒术攻击 = (有效咒力总量 * 0.2 * 咒力操纵精度 + 术式基础强度) * 咒力输出程度 * 伤害补正;

const 物理攻击 = (总肉体值 * 0.5 * 天与补正 + 武艺值) * 武艺输出比例 + 咒具值;

const 总BP = Math.max(总肉体值, 术式强度, 咒术攻击, 物理攻击);
```

### 测试策略

- 单元测试：
  - 武艺阶段映射。
  - 术式潜力等级映射。
  - 大脑重置惩罚映射。
  - 特殊体质补正。
- 集成测试：
  - 初始化变量 parse 后能得到完整派生字段。
  - 五条悟案例能得到接近框架示例的结果。
  - 空集合不出现 `待初始化`。
- 性能测试：
  - transform 仅进行常量级计算，无递归重型运算。
- 兼容性测试：
  - 前端仍可读取 `战斗面板.总BP`、`总肉体值_BPA`、`术式强度_BPB`、`最大生命值`、`当前血量`。

### 风险应对计划

- 风险项 1：特殊体质名称不统一。
  - 应对：变量更新规则固定名称：`天与咒缚（纯肉体）`、`天与咒缚（咒力异化）`、`六眼`、`黑闪眷顾者`、`九相图`、`受肉体`。
- 风险项 2：派生字段被 AI 直接写入导致冲突。
  - 应对：规则中标明派生字段禁止直接修改，schema transform 最终覆盖派生结果。
- 风险项 3：前端旧公式展示与新公式不一致。
  - 应对：保留前端字段，优先让数据正确；若后续需要，可再改前端展示细节。
