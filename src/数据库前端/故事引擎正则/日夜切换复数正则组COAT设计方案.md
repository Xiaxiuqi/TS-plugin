# 日夜切换复数正则组 COAT 设计方案

## 插件开发方案

### 选定的实现路径

采用“顶层主题开关 + CSS 变量令牌 + 复数子正则共享作用域”的方案。

核心判断：

- 单条正则，如变量更新面板，可以用本条替换内容内部的 `details[open] ~ .shell` 完成局部日夜切换。
- 复数正则组，如故事引擎悬浮球，外层正则、事件条正则、NPC 卡片正则是分阶段替换生成的多个片段。它们要统一切换，不能让每个子片段各自拥有独立开关，否则事件卡、NPC 卡、外层面板会出现不同主题混杂。
- 因此复数正则组必须把主题状态放在“外层正则生成的共同祖先容器”中，子正则只输出结构类名，不输出主题开关，也不直接写死颜色。
- 主题切换不使用 JavaScript，使用 `<details class="theme-state">` 的 `[open]` 状态作为 CSS 状态源，通过兄弟选择器 `theme-state[open] ~ panel` 或外层容器内 CSS 变量覆盖来同步整组样式。
- “暗色”统一改名为“暗色”，作为夜间模式的视觉风格名称。

## COAT 流程

### 1. 联想扩展结果

#### 技术上下文联想

- 当前环境是酒馆正则替换渲染，替换内容以 HTML/CSS 片段为主。
- 多数正则文件不是单组件，而是多条正则流水线：
  - 外层面板正则负责包裹整体容器、主 CSS 和捕获组布局。
  - 事件正则负责将事件行替换成子卡片。
  - NPC 正则负责将 `<npc_driver>` 替换成 NPC 子模块。
  - BP 系统也可能存在外层卡片、目标卡片、等级/稀有度处理等复数规则。
- 正则替换不保证后续子正则能修改外层已有 CSS，因此主题架构必须在第一条“外层正则”里一次性定义完整 CSS。
- 子正则输出的 HTML 必须处于外层容器内部，才能继承外层 CSS 变量和主题状态。
- 不应依赖浏览器全局状态、`localStorage`、脚本注入或事件监听；否则在消息渲染、安全策略、酒馆正则阶段都可能不稳定。
- `<details>` 与 CSS 选择器是当前最稳定的无脚本交互方式。
- 每条消息生成的 DOM 是独立的，因此默认每条消息独立切换主题；如果要全局同步所有消息，需要脚本或宿主级设置，不适合纯正则方案。

#### 功能需求联想

- 主功能：把米白与暗色两套视觉统一进一个正则组。
- 辅助功能：
  - 增加“日 / 夜”切换按键区。
  - 按键状态要影响整组正则生成的所有子模块。
  - 标题、图标、页脚文案可随主题切换。
  - 子卡片不重复定义主题逻辑。
- 用户交互流程：
  1. 用户看到默认日间 / 米白主题。
  2. 点击主题按钮。
  3. 按钮切换为夜间 / 暗色状态。
  4. 外层面板、事件卡、NPC 卡、战斗区、页脚统一变为暗色。
  5. 再次点击恢复日间。
- 数据流设计：
  - 输入：AI 输出文本中的结构化标签与纯文本块。
  - 正则阶段 1：外层正则捕获主结构并生成共同根容器。
  - 正则阶段 2：事件正则只生成 `.se-event-card` 等子结构。
  - 正则阶段 3：NPC 正则只生成 `.se-npc-card` 等子结构。
  - 输出：同一根容器内的完整主题化 HTML。
- 边界情况：
  - 如果子正则在外层正则之前执行，子片段可能无法进入外层容器的目标插槽，因此必须强调执行顺序。
  - 如果事件正则作用范围过宽，可能替换到外层之外的普通列表项。
  - 如果多个故事引擎面板在同一消息中出现，主题开关应各自独立，不互相影响。
  - 如果用户只复制子正则，不复制外层正则，子卡片将失去完整样式。

#### 实现约束联想

- 命名必须加组件前缀，避免污染其他正则：
  - 故事引擎建议使用 `.story-engine-orb-widget.se-dual-ui`。
  - 变量更新建议使用 `.var-update-widget.var-update-dual-ui`。
  - BP 面板建议使用 `.bp-widget.bp-dual-ui`。
- CSS 变量必须统一命名：
  - 背景：`--panel`、`--panel-2`、`--card`、`--row`
  - 文字：`--text`、`--muted`、`--faint`
  - 重点色：`--gold`、`--blue`、`--cyan`、`--rose`
  - 阴影：`--shadow`、`--shadow-soft`
  - 线条：`--line`
- 子正则不得写 `style=""` 中的固定颜色，优先输出类名与内容。
- 暗色不是单独文件，而是夜间变量覆盖。
- 正则组说明中必须保留执行顺序。
- 每个合并版正则文件应写清：
  - 哪条是外层主题宿主。
  - 哪些是子正则。
  - 子正则是否依赖外层容器。

### 2. 候选方案对比

#### 方案 1：每个子片段各自内置主题开关

```text
方案 1: 子片段自治切换
├── 实现路径:
│   - 每条正则替换内容都包含一套日夜 CSS 与一个开关
│   - 事件卡、NPC卡、外层面板分别切换
│   - 数据存储结构为每个片段自己的 details[open]
├── 技术栈选择:
│   - 纯 HTML/CSS
│   - details/summary
│   - CSS 兄弟选择器
├── 开发工作量预估:
│   - 核心代码量：每个正则片段增加 80~200 行
│   - 测试代码量：低
│   - 集成调试时间：中
└── 风险点:
    - 技术风险：主题状态无法统一
    - 兼容性风险：多开关造成视觉割裂
    - 性能风险：重复 CSS 体积大
```

#### 方案 2：外层主题宿主 + 子正则继承变量

```text
方案 2: 外层统一状态切换
├── 实现路径:
│   - 第一条外层正则生成共同根容器、主题开关、完整 CSS 变量
│   - 子正则只输出类名结构，不输出主题逻辑
│   - 外层 details[open] 通过兄弟选择器切换 CSS 变量
│   - 子模块全部通过 var(...) 继承主题
├── 技术栈选择:
│   - 纯 HTML/CSS
│   - details/summary
│   - CSS 自定义属性
│   - 组件前缀类名
├── 开发工作量预估:
│   - 核心代码量：外层 CSS 增加 150~350 行，子正则少量改名或去内联样式
│   - 测试代码量：中
│   - 集成调试时间：中
└── 风险点:
    - 技术风险：必须严格保证正则执行顺序
    - 兼容性风险：旧浏览器 CSS 变量支持问题，但现代酒馆环境通常可用
    - 性能风险：较低，CSS 只集中定义一次
```

#### 方案 3：全局 CSS 类 + JavaScript 按钮同步所有面板

```text
方案 3: 脚本驱动全局主题
├── 实现路径:
│   - 注入按钮与脚本
│   - 点击按钮后给 document.body 或消息容器添加主题 class
│   - 所有正则面板读取全局 class
│   - 可用 localStorage 持久化
├── 技术栈选择:
│   - HTML/CSS/JavaScript
│   - DOM 事件监听
│   - localStorage 可选
├── 开发工作量预估:
│   - 核心代码量：100~250 行
│   - 测试代码量：中高
│   - 集成调试时间：高
└── 风险点:
    - 技术风险：酒馆正则替换阶段可能不允许或不稳定执行脚本
    - 兼容性风险：消息重复渲染时事件绑定重复
    - 性能风险：全局查询和重复脚本可能膨胀
```

### 4. 回溯记录

- 初始试作的 `变量更新美化正则-日夜切换版` 属于“单组件局部开关”，适合单条正则。
- 但用户指出剩余大多是复数正则组，必须重新评估。
- 回溯触发原因：
  - 单组件开关无法直接推广到复数正则组。
  - 如果事件条、NPC 卡都自己带开关，会导致同一故事引擎内主题不同步。
- 回溯后选择方案 2：
  - 保留无脚本优势。
  - 将主题状态提升到外层正则。
  - 子正则只负责结构输出。
  - 外层 CSS 统一控制所有子模块。

## 最终技术方案

### 架构设计

#### 核心模块

- 外层主题宿主模块：
  - 负责输出共同根容器。
  - 负责输出日夜切换按钮区。
  - 负责输出完整 CSS 变量与日夜覆盖。
  - 负责包裹所有 `$捕获组` 插槽。
- 子正则结构模块：
  - 事件条模块：只输出 `.se-event-card`、`.se-event-top`、`.se-event-name`、`.se-event-rate`、`.se-event-text`。
  - NPC 模块：只输出 `.se-npc-card`、`.se-npc-name`、`.se-chip`、`.se-npc-line` 等。
  - BP 目标模块：只输出 `.bp-target-card`、`.bp-row`、`.bp-meter` 等结构类。
- 主题令牌模块：
  - 日间令牌写在外层容器默认变量。
  - 夜间令牌写在 `.theme-state[open] ~ .theme-shell` 或 `.theme-state[open] ~ .panel` 覆盖块。
- 文案切换模块：
  - 同一位置放两份短文案：
    - `.mode-day`
    - `.mode-night`
  - 夜间时通过 CSS 显隐切换。

#### 数据流

```text
AI 原始输出
  ↓
外层正则捕获完整结构
  ↓
输出根容器 + 主题开关 + 主 CSS + 插槽区域
  ↓
事件/NPC/BP 等子正则在插槽内容中继续替换
  ↓
子结构继承外层 CSS 变量
  ↓
用户点击主题按钮
  ↓
details.theme-state 切换 open 状态
  ↓
CSS 变量覆盖整组子模块
```

#### 接口定义

##### 外层正则输出要求

```html
<div class="xxx-widget xxx-dual-ui">
  <style>
    /* 全部主题变量与所有子模块样式 */
  </style>

  <details class="xxx-theme-state">
    <summary class="xxx-theme-button">...</summary>
  </details>

  <details class="xxx-shell">
    ...
    <div class="xxx-child-slot">
      $子内容捕获组
    </div>
    ...
  </details>
</div>
```

##### 子正则输出要求

```html
<div class="xxx-child-card">
  <div class="xxx-child-title">$1</div>
  <div class="xxx-child-body">$2</div>
</div>
```

##### 禁止项

```html
<!-- 不推荐：子正则自带主题开关 -->
<details class="child-theme-state">...</details>

<!-- 不推荐：子正则写死颜色 -->
<div style="background:#171b22;color:#ede7dc">...</div>

<!-- 不推荐：子正则重复注入整套 style -->
<style>...</style>
```

### 实现步骤

1. 先处理单条正则组件：
   - 用 `变量更新美化正则-日夜切换版` 验证 `details[open] + CSS变量` 方案可行。
2. 处理故事引擎复数正则组：
   - 外层正则改为 `.story-engine-orb-widget.se-dual-ui`。
   - 外层正则包含主题按钮、日间变量、夜间变量覆盖、所有子模块样式。
   - 事件条和 NPC 卡片保持现有替换结构，但不要独立主题。
3. 处理 BP 复数正则组：
   - 把所有等级颜色、卡片背景、条形图颜色抽象为变量。
   - 外层正则提供统一主题按钮。
   - 目标卡片正则只输出结构与等级类名。
4. 处理状态栏 HTML 前端：
   - 状态栏是完整前端，允许使用按钮和脚本/局部状态，但若要和纯正则一致，仍优先采用同样的局部主题根容器。
5. 统一命名：
   - “米白”作为日间。
   - “暗色”作为夜间。
   - 文件名建议采用 `日夜切换版`。
   - 不再使用“暗色”作为最终公开主题名。

### 测试策略

#### 单元测试

- 检查外层正则捕获组数量不变。
- 检查替换内容仍包含原占位符：
  - 故事引擎外层：`$1` 到 `$14`
  - 事件条：`$1`、`$2`、`$3`
  - NPC 卡：`$1` 到 `$11`
  - 变量更新：`$2`
- 检查所有子正则不包含独立主题按钮。

#### 集成测试

- 用一段含多个事件、多个 NPC、战斗驱动、最终修正的故事引擎样例文本测试：
  - 默认日间渲染。
  - 点击切换后外层、事件、NPC、战斗、页脚全部变夜间。

  - 折叠主题按钮不影响故事引擎面板展开状态。
- 用多条消息测试：
  - 第一条切夜间，第二条仍默认日间。
  - 每条消息状态互不污染。

#### 性能测试

- 检查一条消息中多个故事引擎面板时 CSS 不出现全局污染。
- 控制外层 CSS 体积，不在子正则重复注入 `<style>`。
- 避免复杂全局选择器，全部以根容器类名前缀限定。

#### 兼容性测试

- 桌面端 Chrome / Edge。
- 酒馆内嵌 WebView。
- 移动端窄屏：
  - 事件卡从双列变单列。

### 风险应对计划

#### 风险 1：复数正则执行顺序错误

- 应对措施：
  - 文档中保留“必须严格按顺序创建并启用”。
  - 外层正则编号 `01`。
  - 子正则编号 `02`、`03`。
  - 子正则不依赖后续 CSS，只依赖外层 CSS。

#### 风险 2：主题切换只影响外层，不影响子卡片

- 应对措施：
  - 子卡片所有颜色必须使用外层 CSS 变量或继承变量。
  - 子正则禁止硬编码颜色。
  - 子卡片必须在外层 `.se-shell` 或 `.se-panel` 内。

#### 风险 3：单个 `<details>` 既用于折叠又用于主题，状态冲突

- 应对措施：
  - 使用两个独立 `<details>`：
    - `.theme-state` 专门存主题。
    - `.shell` 或 `.panel-state` 专门负责展开内容。
  - 二者为同级元素，使用兄弟选择器联动样式。

#### 风险 4：想要全局同步所有面板主题

- 应对措施：
  - 纯正则方案默认不做全局同步。
  - 如后续必须全局同步，再单独设计脚本插件方案，不混入正则组基础版。

#### 风险 5：编辑器误报 CSS 错误

- 应对措施：
  - 这些文件是 Markdown 中嵌 HTML/CSS 代码块，编辑器若按 CSS 解析会误报。
  - 以实际酒馆渲染和浏览器解析为准。
