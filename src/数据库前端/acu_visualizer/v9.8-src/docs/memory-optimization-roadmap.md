# ACU Visualizer 最大化内存优化路线图

本路线图用于在不影响原功能和 CSS 的前提下，规划最大程度降低 ACU Visualizer 内存占用的实施顺序。

## 1. 总原则

- 原插件运行文件默认不动：
  - `public/acu_visualizer/acu_visualizer.js`
  - `public/acu_visualizer/acu_visualizer-test.js`
- 所有优化先在 `public/acu_visualizer_test/` 测试版中规划和验证。
- CSS 相关优化必须先复制、对照、hash 验证，不直接替换原注入逻辑。
- 功能相关优化必须保留原行为、原 class、原 localStorage key、原数据库 API 调用。
- 每个优化点必须具备回滚方案。

## 2. L0：零风险文档与检测层

目标：建立可量化基线和风险控制，不改变运行逻辑。

### 内容

1. 建立内存基线测量清单。
2. 建立 CSS 零破坏检查清单。
3. 建立变更记录制度。
4. 设计测试场景与指标。
5. 确定每个优化点的 fallback 和回滚方式。

### 风险

无运行风险，仅文档修改。

### 交付物

- `docs/memory-baseline-checklist.md`
- `docs/css-safety-checklist.md`
- `docs/memory-optimization-roadmap.md`
- `docs/change-log.md`

## 3. L1：泄漏治理层

目标：最大优先级处理重复初始化、事件监听、弹窗 DOM、菜单回调造成的长期泄漏。

### 优化点

1. 统一 `destroy()`：
   - 清理测试版事件监听。
   - 移除测试版弹窗和菜单。
   - 清空临时 Set/Map。
   - 移除跨 document 的关闭监听。
2. 事件命名空间：
   - 统一 `.acu` 或 `.acuTest`。
   - 重新初始化前统一 off。
3. 弹窗管理器：
   - 设置弹窗。
   - 历史弹窗。
   - 编辑弹窗。
   - 快捷选项弹窗。
   - 确认弹窗。
4. 通知队列释放：
   - 通知消失后从队列删除。
   - destroy 时清空通知。

### 安全要求

- 不改 CSS。
- 不改 DOM class。
- 不改变任何按钮行为。
- 仅清理无效引用。

### 预期收益

长期使用和热重载后的内存增长明显降低。

## 4. L2：常驻内存下降层

目标：降低脚本长期持有的数据量。

### 优化点

1. 样式按需注入：
   - 搜索 CSS 按需。
   - 快捷选项 CSS 按需。
   - 弹窗 CSS 可分组。
2. 运行态 key 压缩：
   - 表名映射为短 id。
   - diff key 从长表名字符串改为短 key。
3. 状态过期清理：
   - 当前不存在的表清理 diffMap。
   - 保存成功后清理 userEditMap。
   - 取消删除后清理 pendingDeletes。
4. 快照轻量化：
   - 完整快照改为表级 hash、行级 hash。
   - 保留旧格式兼容读取。
5. 历史记录瘦身：
   - 分片存储。
   - 单格上限。
   - 全局上限。
   - 过长值摘要化。

### 安全要求

- 旧数据格式必须兼容读取。
- 不删除用户数据，除非用户确认清理。
- 高亮状态不能被错误清空。

### 预期收益

降低 localStorage、闭包缓存和运行态 Set/Map 的常驻占用。

## 5. L3：峰值内存下降层

目标：降低刷新、渲染、搜索、弹窗打开时的瞬时内存峰值。

### 优化点

1. 当前 tab 懒渲染：
   - 只渲染激活表。
   - 非激活表仅保留占位。
2. 当前页渲染：
   - 保持分页。
   - 不为不可见页创建 DOM。
3. 局部 patch：
   - 表头不变时复用。
   - 只更新 tbody。
   - 更进一步只 patch 变化单元格。
4. 搜索限制：
   - debounce。
   - 只搜索当前页。
   - 限制最大高亮数。
   - WeakMap 保存原文。
5. 写入节流：
   - 滚动位置节流。
   - 分页状态合并写入。
   - 相同 JSON 跳过写入。

### 安全要求

- 视觉结果必须一致。
- 分页、标签、编辑、拖拽、保存行为必须一致。
- 必须保留 fallback 到原渲染路径。

### 预期收益

大表格刷新和搜索时的内存峰值显著降低。

## 6. L4：最大收益重构层

目标：在 L1-L3 稳定后，实施高收益但高风险的高级优化。

### 可选优化

1. 虚拟滚动：
   - 仅渲染视口附近行。
   - 适合极大表格。
2. Worker diff：
   - 大型 JSON diff 放到 Worker。
   - 减少主线程阻塞。
3. IndexedDB 历史库：
   - 历史记录从 localStorage 移出。
   - 适合大量历史。
4. CSS 构建压缩：
   - 构建期压缩 CSS。
   - 运行时按需注入。

### 安全要求

- 必须 feature flag。
- 必须原路径 fallback。
- 必须先有独立测试版 bundle。
- 必须有视觉与行为对照。

## 7. 推荐顺序

1. 完成 L0 文档与测量。
2. 实施 L1 泄漏治理。
3. 实施 L2 状态、快照、历史瘦身。
4. 实施 L3 懒渲染与局部 patch。
5. 评估 L4 是否必要。

## 8. 回滚策略

每个阶段必须可单独回滚：

- L1 回滚：关闭 destroy/事件命名空间改造，恢复原事件绑定。
- L2 回滚：继续读取旧快照与旧历史；新格式只作为增强，不删除旧数据。
- L3 回滚：feature flag 切回完整渲染。
- L4 回滚：禁用虚拟滚动、Worker、IndexedDB 路径。
