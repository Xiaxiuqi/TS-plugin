  - 当前 main.py 第312-327行主循环：for i in range(count): if register_one(...): success += 1。单账号失败立即进入 i+1，没有"当前序号下重试"概念

  - 需求3（用户字面）：单账号失败后在当前序号下重试，比如"第一次因为超时或邮箱黑名单"，重试时打印仍是"第 1/N 个"，而不是直接进入第 2/N 个
  - 隐含需求：account-level 重试必须有最大次数和等待间隔，避免无限循环和触发限流；这是 outer retry，独立于 register_one 内部已有的 inner email retry

  - 挑战1：Civic Cookie Hub 弹窗渲染时机不确定。按 Civic 的常见行为，<div id="ch2"> 在 DOMContentLoaded 后再异步加载，2.5s 固定 sleep 偶尔来不及；用 wait_for_selector 等"按钮可见且可点"才是稳的做法
  - 挑战2：弹窗有时有淡入动画，按钮 in DOM 但 is_visible() 在动画期间可能返回 False；需要短间隔多次重试点击
  - 挑战3：account-level 重试的语义边界——"超时或邮箱黑名单"这两种失败都对应 register_one raise 出来的异常。当前 register_one 把所有异常都吞在 except 里返回 False，外层无法区分类别；好消息是用户要求的就是"任何失败都在当前序号下重试"，所以不需要区分异常类别，只看 register_one 返回值即可
  - 挑战4：account-level 重试要求每次都开新浏览器（fresh state），这正好和 with JetBrainsBrowser(...) 在 register_one 内部的现有结构匹配——不需要改 register_one 本身

  - 改动2（jetbrains_register.py::accept_cookies_on_notifications）：把 "fixed sleep + 单次尝试" 改成 "等弹窗容器出现 + 最多 N 次重试点击"：
    * 已知容器选择器列表：#ch2-dialog, .ch2, #onetrust-banner-sdk, #cookie-banner, [role="dialog"][aria-label*="cookie" i]
    * 用 page.wait_for_selector(combined, state="visible", timeout=8_000) 等任一容器可见
    * 找到容器后循环最多 5 次，每次先 _dismiss_cookie_banner，再等 1.2 秒检查弹窗是否消失；消失即成功
    * 整体仍用 try/except 包裹失败不影响成功判定
  - 改动3（main.py main 主循环）：把 for i in range(count) 改成 while account_idx < count + 内层 attempt 循环：
    * 同一序号下 attempt 从 0 到 max_account_retries
    * 失败时打印 "[i] 第 X/N 个账号重试 m/M..."，等 account_retry_wait 秒
    * 成功 break 进入下一序号；重试用尽也进入下一序号（避免一个账号卡死整个批次）
    * 新增 argparse: --max-account-retries（默认 2，即首次 + 2 次重试 = 共 3 次尝试）和 --account-retry-wait（默认 5.0）
